SeLion PageYaml

YAML pages in the SeLion world are equivalent to object repositories. They help in abstracting away all of the locator information, the location strategy to be used to identify an element, the challenges with handling localization etc., and help keep the test code clean.

They contain the following information with respect to a given page and represent a particular page as an Object, so that test code can leverage them to interact with the specific page.

  • The Java class from which the newly generated page should extend itself from.
  • The title of the page
  • The set of elements, the ways of locating them (lets call this as locating strategies) and the actual locators themselves.
  • A set of validators that can be used to trigger validations on a given page once it is loaded as a result of an operation on the page.

PageYaml Contents

A YAML page that represents page related information, would essentially have the below mentioned contents in it:

  • baseClass - The value for this key represents the fully qualified package name of a class that will serve as the base class from which all common abilities that any page in the SeLion world would possess. If your page is totally new and is not customizing an existing page (for example, a login page which extends a header page ), then you should be specifying the value for this key as com.paypal.selion.testcomponents.BasicPageImpl because by default this is the class within SeLion that provides for all the basic operations that are associated with a given page, including logic pertaining to instantiation of the page as well.
  • pageTitle - The value for this key is a set of key/value pairs wherein you would like to capture the page titles for various different combinations (locales, for example).
  • elements - The value for this key is a set of key/value pairs that represents the below information pertaining to html elements in the page.
    • The name to be given to identifying the locator. This name would end up serving as the data member in the generated class which represents this element. The rules for naming the locator are as below :
      • <prefix>Button Is a Button html element.
      • <prefix>CheckBox Is a Check Box html element.
      • <prefix>DatePicker Is a Date Picker html element.
      • <prefix>Form Is a Form html element.
      • <prefix>Image Is a Image html element.
      • <prefix>Label Is a Label html element.
      • <prefix>Link Is a Link html element.
      • <prefix>RadioButton Is a Radio Button html element.
      • <prefix>List Is a Select List html element.
      • <prefix>Table Is a Table html element.
      • <prefix>TextField Is a Text Field html element.
      • <prefix>Container Is a Container html element.

      Here prefix is a meaningful name that you would give to your element, mapping it to the html element on the page.

    • locators: - This key represents the set of values that map to the element name.
    • Locator - Values for the locators which represents a key/value pair of locale and the actual locator for the html element. Example: US: "//div[@id='apiOption1']/p[3]/a"

      which means that this particular link can either be found using the xpath //a[contains(@href,
        'cmd=_profile-api-add-direct-access')] or using the xpath //a[contains(@href,
        'cmd=%5fprofile%2dapi%2dadd%2ddirect%2daccess')]

  • Notice that only for the element requestAPICredentialsLink we are having locators for 2 locales
      viz., US and FR. This is another salient feature of SeLion. For specific elements alone you can have
      locators defined for specific locales. So for the same locale say FR since the element viewAPICertificateLink
      doesn't have a locator for FR SeLion will default to using the locator specified for the locale US
      while working with the country FR.
